6. Debugging with Network Dev Tool

Use Chrome’s Network Dev Tool to debug failed requests, inspect response data, and analyze headers and timing. Access it via F12 (Windows) or Command + Option + I (Mac), then select the Network tab.

  • Debugging Tips: Look for red status codes (e.g., 404 for Not Found), check the “Response” tab for error messages, and use “Headers” to inspect metadata like Content-Type or Cache-Control.
  • Common Issues: Incorrect URLs, server errors (e.g., 500), CORS restrictions.

Example: Debugging a Bad Request

Try a request to an invalid URL and inspect it:


// Attempt to fetch data from an invalid endpoint
fetch("https://swapi.co/api/invalid")
    .then(res => res.json())
    .catch(err => console.log("Error:", err)); // Logs network or parsing error

Open Dev Tools (F12 on Windows, Command + Option + I on Mac), go to the Network tab, and run the code. You’ll see a 404 status code. Check the “Response” tab for the error message (e.g., “Not found”) and the “Headers” tab for details like:

  • Status: 404 Not Found
  • Content-Type: application/json
  • Cache-Control: no-cache (controls caching behavior)
  • Access-Control-Allow-Origin: * (allows cross-origin requests)

Why Examine Headers?

Headers provide metadata about the request and response, helping you debug issues:

  • Content-Type: Confirms the response format (e.g., application/json).
  • Cache-Control: Indicates if the response is cached or fresh.
  • Access-Control-Allow-Origin: Checks for CORS permissions (e.g., blocked requests).
  • Authorization: Verifies authentication tokens if required.

Open the Network tab (F12 on Windows, Command + Option + I on Mac), find the failed request to invalid, and note the status code (404) and error message in the “Response” tab. Then, check the “Headers” tab and identify the Content-Type, Cache-Control, and Access-Control-Allow-Origin values.
